Overview

What It Does

OSOM-AI is a natural-language-to-SQL (NL2SQL) assistant that lets business users query a PostgreSQL database in plain English. It translates conversational questions into validated SQL queries, executes them safely, and returns structured results or visualizations — no SQL knowledge required.

At its core is a semantic modeling layer that describes the database schema — tables, columns, relationships, and business terminology — as semantically rich text chunks. These chunks are indexed in ChromaDB and retrieved at query time using Vanna.ai's RAG framework, giving the LLM the precise schema context it needs to generate accurate SQL.

The system features LLM intent routing to distinguish between conversational messages, schema exploration queries, and SQL-generation requests. When generated SQL fails execution, a self-correcting repair loop feeds the error back to the LLM for iterative correction — typically resolving syntax and logic errors within 2-3 iterations.

A read-only query safety layer parses and validates every generated SQL statement before execution, blocking mutations (INSERT, UPDATE, DELETE, DROP) and enforcing row-limit guards. The whole system is exposed via a FastAPI REST endpoint with an automated accuracy evaluation harness that benchmarks NL→SQL correctness against a held-out question set.

Features

Key Components

Semantic Modeling Layer
Database schema described as semantically enriched text chunks — tables, columns, foreign keys, and business glossary — giving the LLM ground-truth context instead of raw DDL noise.
ChromaDB RAG Retrieval
Schema chunks are embedded and stored in ChromaDB. At query time, the most relevant schema context is retrieved via semantic similarity search and injected into the LLM prompt — ensuring precise, scoped SQL generation.
LLM Intent Routing
A routing layer classifies each user message — conversational chitchat, schema exploration, or SQL-answerable query — and dispatches to the appropriate handler before any SQL generation is attempted.
Self-Correcting SQL Repair
When generated SQL fails execution, the database error is fed back to the LLM with the original intent for iterative self-correction, typically resolving syntax and logic errors within 2-3 repair cycles.
Read-Only Safety Validation
Every generated SQL statement is parsed and validated before execution. Mutation operations (INSERT, UPDATE, DELETE, DROP, TRUNCATE) are blocked; row-limit guards prevent accidental full-table scans.
Automated Accuracy Harness
A held-out evaluation set of natural-language questions with expected SQL outputs benchmarks end-to-end NL→SQL accuracy, tracking exact-match and execution-equivalence scores across schema changes.
